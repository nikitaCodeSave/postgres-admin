Помогаю создать спецификацию новой фичи в backlog проекта.

## Контекст

Спецификации фич хранятся в `docs/backlog/features/` с форматом имени: `[priority]-[short-name].md`

**Приоритеты**:
- **P0** - Критично, must have
- **P1** - Высокий приоритет
- **P2** - Средний приоритет
- **P3** - Низкий приоритет, nice to have

## Процесс создания спецификации

1. **Задать вопросы**:
   - Название фичи? (короткое, 2-5 слов)
   - Приоритет? (P0 / P1 / P2 / P3)
   - Какую проблему решает эта фича?
   - Какое предлагается решение?
   - Какую ценность приносит пользователям?
   - Оценка сложности? (S / M / L / XL)

2. **Создать файл спецификации**:
   - Использовать шаблон: `docs/backlog/features/feature-template.md`
   - Имя файла: `docs/backlog/features/[priority]-[kebab-case-name].md`
   - Заполнить секции на основе ответов
   - Установить статус "Planned"
   - Установить сегодняшнюю дату: !`date +%Y-%m-%d`

3. **После создания сообщить**:
   - Путь к файлу
   - Следующие шаги: детализировать технические требования, создать mockups, уточнить оценку

## Примеры хороших названий

✅ **Хорошие примеры**:
- `P1-database-browser.md` - Database Browser UI
- `P2-query-history.md` - Query History Tracking
- `P0-user-authentication.md` - User Authentication System
- `P1-schema-designer.md` - Visual Schema Designer

❌ **Плохие примеры** (слишком мелко):
- `P2-add-button.md` - лучше как improvement
- `P3-fix-typo.md` - это баг, не фича

## Критерии для фичи

✅ **Это фича, если**:
- Добавляет НОВУЮ функциональность
- Требует существенной разработки
- Приносит явную ценность пользователям
- Заслуживает отдельной спецификации

❌ **Это НЕ фича, если**:
- Улучшение существующей функции → используй `/create-improvement`
- Исправление бага → используй `/create-bug`
- Мелкое изменение → просто сделай без спеки

**Важно**: Если фича кажется слишком маленькой, я предложу создать improvement вместо feature.
